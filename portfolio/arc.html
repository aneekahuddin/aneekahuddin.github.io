Advanced Replacement Cache (ARC)

Problem: Develop a cache that works for applications that don’t know whether frequency or recency should be the metric of use.

Our Approach

First, we figured out how the cache worked from the paper.

We sought to understand the different abstractions and how that would translate to different data structures. Adaptive Replacement cache utilizes both LRU and LFU caching systems. To determine the ratio of LRU to LFU, a calculation is done based on how many times an element enters a cache and how large the “ghost entries” (items that were no longer in the cache) of each cache was.

Implementation

Both my partner and I had worked on another caching project before with simpler caches such as FIFO, LRU, and LFU. We used that knowledge to consider how we would organize our data structures and what classes could be helpful. Eventually we decided to use slices and dicts to keep track of the different lists in the cache, and the number of entries in each which would change the ratio between the LRU and LFU sides of the cache. (Hence, adaptive replacement).

Testing

After developing a working implementation, we decided to test for correctness. We checked that each of the four major actions and all of the subcases of each action happened in accordance with the paper. Further, we tested out each path in our code, edge cases (empty caches, etc.), and then stress tested (random calls with random users). Once we were certain that our code was running properly, we cleaned it up to make it more readable.
